Microsoft 停用部分 Claude Code license：团队要把 AI 工作流做成工具无关Microsoft’s Claude Code Pullback Shows Why Agent Workflows Must Be Portable

企业会换工具，AI 编程流程要做成可迁移资产。

Enterprises will switch tools; make your AI coding workflow portable.

Microsoft 计划在 2026 年 6 月底前让 Experiences + Devices 团队逐步停用大部分 Claude Code license，并把开发者迁移到 GitHub Copilot CLI。对日常用 Claude Code/Codex 的工程师来说，这提醒我们：AI 编程工作流必须工具无关，不能绑死在单一供应商或单一 CLI 上。

可落地要点

  • 企业内部会因为成本、采购、安全、产品战略切换 agent 工具；个人和团队应该把关键上下文沉淀到 repo，而不是依赖某个工具的会话记忆。
  • 用 AGENTS.md、CLAUDE.md、CODEX.md、脚本和测试命令固化工作方式，让 Claude Code、Codex、Copilot CLI、Cursor Agent 都能读懂同一套任务入口。
  • 对团队而言，迁移成本最高的不是模型能力，而是隐含工作流：常用 prompt、权限策略、MCP、hook、review checklist、测试命令、提交规范。
  • 可以建立“agent 可移植性冒烟测试”：同一个 issue，分别让两个工具跑一遍，看是否都能找到入口、运行测试、产出可 review 的 diff。
  • 采购或平台策略变化不是坏事，但它会惩罚“只会点某个工具按钮”的团队，奖励把工程流程脚本化、文档化、可验证化的团队。

现在能做的一步

在每个 repo 根目录放一个工具无关的 AGENTS.md，把入口命令和交付要求写清楚：

# Agent Instructions

## Setup
- Use the existing package manager lockfile.
- Install dependencies with `pnpm install` if `node_modules` is missing.

## Verify
- Run `pnpm test` for unit tests.
- Run `pnpm lint` before finalizing UI or TypeScript changes.
- For backend changes, run `pnpm test:integration` when touching API contracts.

## Working Rules
- Keep changes scoped to the requested behavior.
- Do not rewrite unrelated files.
- Prefer existing helpers and patterns over new abstractions.
- Final response must include changed files and verification results.

再加一个可复制给任意 agent 的迁移测试提示词：

Read AGENTS.md first. Make the smallest change needed for this issue, run the documented verification commands, and report any command that cannot run. Do not rely on tool-specific memory or prior chat context.

如果这个提示词在 Claude Code、Codex 和 Copilot CLI 中表现差异很大，优先修 repo 文档和脚本，而不是只调 prompt。

Actionable takeaways

  • Enterprises can switch agent tools for cost, procurement, security, or product-strategy reasons; keep critical context in the repo, not only in one tool's chat memory.
  • Use AGENTS.md, CLAUDE.md, CODEX.md, scripts, and test commands to define the workflow so Claude Code, Codex, Copilot CLI, and Cursor Agent can all operate from the same entry point.
  • The hardest migration cost is not model capability; it is hidden workflow state: common prompts, permission policies, MCPs, hooks, review checklists, test commands, and commit conventions.
  • Build an “agent portability smoke test”: run the same issue through two tools and check whether both can find the entry point, run tests, and produce a reviewable diff.
  • Procurement or platform shifts are not inherently bad, but they punish teams that only know one tool's UI and reward teams that script, document, and verify their engineering process.

Superset：面向 agent 时代的 IDE，用 worktree 并行调度 Claude Code/CodexSuperset Turns CLI Coding Agents into Parallel Worktree-Based Workflows

多 agent 并行开发正在变成 IDE 的核心能力。

Parallel multi-agent development is becoming a core IDE capability.

Superset 是一个开源 agentic IDE，用本地 git worktree 同时运行 Claude Code、Codex、OpenCode、Copilot CLI 等 CLI agent。它反映了 AI 编程工具的一个重要方向：从“和一个 agent 聊天”升级为“调度多个隔离 workspace 并行产出 diff”。

可落地要点

  • 核心价值是并行：每个任务放在独立 git worktree，避免多个 agent 互相覆盖文件，也方便逐个 review 和合并。
  • 支持任何 CLI agent，适合团队同时比较 Claude Code、Codex、Cursor Agent、Gemini CLI、Copilot CLI 在同一任务上的表现。
  • 内置 terminal、diff viewer、editor handoff、通知等能力，目标是减少你在多个终端、编辑器和分支之间切换。
  • 适合的任务拆分方式：一个 agent 修 bug，一个 agent 补测试，一个 agent 做迁移脚本，一个 agent 调查性能；不要让多个 agent 同时改同一批文件。
  • 即使不用 Superset，也应该学习它的工作流：git worktree 是多 agent 并行开发的基础设施。
  • 当前公开说明偏向 macOS；从源码运行需要 Bun、Git、GitHub CLI 和 Caddy。

现在能做的一步

先用源码体验 Superset：

git clone https://github.com/superset-sh/superset.git
cd superset

cp .env.example .env
echo 'SKIP_ENV_VALIDATION=1' >> .env

# macOS: Caddy 用于本地开发代理
brew install caddy
cp Caddyfile.example Caddyfile
caddy trust

bun install
bun run dev

如果你暂时不装 Superset，也可以用原生 git worktree 模拟多 agent 并行：

# 在主 repo 外创建两个隔离工作区
git worktree add ../repo-bugfix -b agent/bugfix-login
git worktree add ../repo-tests -b agent/add-auth-tests

# 终端 A
cd ../repo-bugfix
claude

# 终端 B
cd ../repo-tests
codex

给多个 agent 分工时，用明确的文件边界：

Agent A: fix the login redirect bug. Own files under src/auth and src/routes only.
Agent B: add regression tests for login redirect. Own files under tests/auth only.
Do not edit files outside your ownership unless you first report why it is necessary.

Superset is an open-source agentic IDE for running Claude Code, Codex, OpenCode, Copilot CLI, and other CLI agents in local git worktrees. It points to an important shift in AI coding tools: moving from “chat with one agent” to “orchestrate multiple isolated workspaces that produce reviewable diffs in parallel.”

Actionable takeaways

  • The core value is parallelism: each task runs in its own git worktree, so agents do not overwrite each other and each diff can be reviewed and merged independently.
  • It supports any CLI agent, which makes it useful for comparing Claude Code, Codex, Cursor Agent, Gemini CLI, and Copilot CLI on the same class of tasks.
  • Built-in terminal, diff viewer, editor handoff, and notifications aim to reduce switching between terminals, editors, and branches.
  • Good task splits: one agent fixes a bug, one adds tests, one writes a migration script, and one investigates performance. Do not let multiple agents edit the same file set.
  • Even if you do not use Superset, copy the workflow: git worktree is core infrastructure for parallel multi-agent development.
  • The public setup is macOS-oriented; running from source needs Bun, Git, GitHub CLI, and Caddy.

  • 它的核心定位可以理解为“Terraform for coding agents”：用声明式配置启动一组长期存在的 coding agents，而不是手工开多个 terminal、tmux pane、Claude/Codex session。
  • 对同时使用 Claude Code 与 Codex 的工程师，价值在于把“临时 agent 团队”固化为可复用 rig：例如 lead、implementation、QA、reviewer、安全审查各自有角色和上下文边界。
  • pod 概念适合表达共享上下文：同一 pod 内 agent 共享外部状态，不同 pod 之间通过明确边通信，避免所有 agent 都读完整仓库。
  • rig discover 可以识别现有 tmux sessions 并生成候选 RigSpec，这适合从现有工作流渐进迁移，而不是一次性重搭。
  • 可借鉴的工作流不是“多 agent 越多越好”，而是把长期重复的协作拓扑固定下来：例如每个 PR 都跑 Claude + Codex 双 reviewer，或让 research pod 并行探索不同方案。
  • 风险也很明确：多 agent 会放大权限、上下文漂移和重复工作。每个 agent 都应有清晰 owner scope、可写文件范围和停止条件。

一个可照搬的多 agent 角色设计模板：

rig: pr-review
pods:
  review:
    shared_memory: true
    agents:
      - name: claude-reviewer
        runtime: claude-code
        role: correctness, edge cases, maintainability
        write: false
      - name: codex-reviewer
        runtime: codex
        role: tests, regressions, patch minimality
        write: false
edges:
  - from: claude-reviewer
    to: codex-reviewer
    purpose: compare findings and deduplicate issues

给 agent 的启动提示词可以这样写：

你是这个 rig 里的 reviewer。只审查当前 PR，不修改文件。
输出必须包含：
- Findings ordered by severity
- File/line reference for each finding
- Reproduction or reasoning
- Suggested minimal fix
如果没有发现问题，明确说明剩余风险和未运行的测试。
