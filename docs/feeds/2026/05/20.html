Codex 0.132.0：Python SDK 认证增强，恢复会话支持 schema 输出Codex 0.132.0: Better Python SDK Auth and Schema-Safe Resumed Automation

Codex 恢复会话也能强制结构化输出。

Codex resumed sessions can now keep enforcing structured output.

Codex 0.132.0 是一次正式功能发布，重点改善 Python SDK、结构化自动化、远程执行器认证和 TUI 启动体验。对 Codex 用户最有价值的是：文本任务更容易脚本化，恢复会话也能继续强制 JSON schema 输出。

  • Python SDK 支持一等认证能力：API key login、ChatGPT browser/device-code flows、account inspection、logout API。团队可以把“本地开发登录”和“CI/API key”分开设计，减少手写 token glue code。
  • 文本型 turn API 现在可以直接传 plain string，适合把内部工具里的 prompt runner 简化掉一层包装。
  • handle-based runs 返回更丰富的 TurnResult，包含 collected items、timing、usage data；这让你可以把 Codex 任务接入成本统计、慢任务告警和回归分析。
  • codex exec resume 支持 --output-schema：长任务中断后可以恢复上下文，同时继续要求结构化 JSON，适合日报、review bot、批量修复管线。
  • TUI 启动更快，原因是 terminal capability probes 改成批量执行；如果你频繁打开 Codex 交互界面，这会减少“启动等待”的摩擦。
  • Remote executor registration 可以复用标准 Codex auth，远程执行器接入少一个单独 credential flow。
  • 图像 fidelity 可以在 app-server turns 中保留，包括 original-resolution local images；做 UI 截图分析、设计稿比对、视觉回归时更可靠。
  • 修复了 goal continuation 在 usage limit 或重复 blocker 下继续循环的问题；这对自动化很关键，因为它能减少 token 被无效重试烧掉。

最值得马上改的是恢复式结构化自动化。示例：

{
  "type": "object",
  "properties": {
    "status": { "type": "string", "enum": ["done", "blocked", "needs_review"] },
    "summary": { "type": "string" },
    "changed_files": { "type": "array", "items": { "type": "string" } },
    "next_step": { "type": "string" }
  },
  "required": ["status", "summary", "changed_files", "next_step"],
  "additionalProperties": false
}

# 新任务：强制输出 schema
codex exec --output-schema codex-result.schema.json \
  '检查当前 diff，找出最可能导致测试失败的点，只输出 JSON。' \
  | tee run-001.json

# 中断后恢复：继续保留上下文，同时仍然强制 schema
codex exec resume <SESSION_ID> \
  --output-schema codex-result.schema.json \
  '继续刚才的分析，补充 changed_files 和 next_step。' \
  | tee run-001-resumed.json

jq -e '.status and .summary and .changed_files and .next_step' run-001-resumed.json

Python SDK 侧可以优先做一次薄封装清理，把“只处理文本”的调用统一成直接传字符串的模式：

# 迁移模式示例：以你项目实际 client/turns API 名称为准
async def run_text_task(client, prompt: str):
    result = await client.turns.create(input=prompt)
    return {
        "items": result.items,
        "usage": result.usage,
        "timing": result.timing,
    }

升级后建议加三类回归测试：--output-schema 字段完整性、resume 后上下文是否保留、usage limit 下是否正确停止。

Decouple SDK and MCP generation from hosted SaaS risk.

HN 讨论暴露了一个很现实的问题：如果 SDK、MCP server 和多语言客户端都依赖托管生成服务，供应商变化会立刻变成工程风险。对 Claude Code / Codex 用户来说，关键启发是把“API 规范到 SDK/MCP”的生成链路纳入仓库和 CI，而不是只依赖外部后台。

  • 先把 openapi.yaml 或 openapi.json 当作一等资产管理：版本化、review、CI 校验、发布前 diff。
  • 为生成物建立可替换后端：Stainless、OpenAPI Generator、Speakeasy、Fern、Kiota、自研模板都可以是候选，但仓库里要保留“如何重新生成”的命令。
  • 不要只备份生成出来的 SDK；还要备份生成配置、patch 脚本、README 模板、分页/鉴权/错误模型的映射规则。
  • 如果你的 API 要给 AI agent 使用，顺手生成 MCP server 或 tool schema；SDK 迁移时把 agent-facing contract 一起测。
  • 用 Codex/Claude Code 做迁移时，任务要拆小：先盘点生成物，再比较候选工具输出，再迁移一个语言，再扩展到全矩阵。
