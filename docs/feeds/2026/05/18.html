Codex 0.131.0 发布：`codex doctor`、统一 `@` picker 与插件/远程工作流升级0.131.0

TUI、插件、远程执行和 Python SDK 都更工程化。

Codex 0.131.0 是一次偏工作流基础设施的更新，覆盖 TUI、@ mention、插件市场、远程执行和 Python SDK。对团队使用 Codex 的工程师，最值得关注的是 codex doctor、统一 picker、插件分享和远程控制能力。

  • TUI 显示更完整：service tier、token usage、permissions/approval mode、workspace roots 和 Markdown tables 更适合长会话排障。
  • @ mention 可在一个 picker 中搜索 files、directories、plugins、skills，减少上下文注入时的切换成本。
  • 插件工作流增加 marketplace CLI、版本感知分享、share checkout 和默认启用 hooks，适合团队沉淀可复用自动化。
  • 远程工作流支持 daemon-managed codex remote-control、runtime enable/disable 和 status reads，适合把 Codex 接进远程开发环境。
  • Python SDK 迁移到 openai-codex / openai_codex，并加入 generated types、concurrent turn routing、approval modes。
  • 现在可做一步：升级后先跑 codex doctor，把诊断结果作为排查 auth、terminal、network、config 的标准入口。

  • @ mentions now search files, directories, plugins, and skills in one picker, backed by app-server plugin metadata. (#19068, #22375)

  • The Python SDK moved to openai-codex / openai_codex, with pinned runtime-generated types, concurrent turn routing, approval modes, and integration coverage. (#21778, #21891, #21893, #21896, #21905, #21910, #22014)
  • Added codex doctor for support-ready diagnostics across runtime, auth, terminal, network, config, and local state. (#22336)

低价 LLM 是否只是阶段性红利：AI 编程要做成本弹性设计Ask HN: Could free/low cost LLMs be a momentary thing?

AI 编程工作流要准备好应对 token 成本波动。

这类讨论的价值在于提醒工程师：AI 编程生产力不能建立在“无限便宜 token”这个单一假设上。Claude Code / Codex 工作流应尽早做模型路由、缓存和任务分层，才能在价格或限额变化时保持可用。

  • 把任务按复杂度分层：简单 grep、格式化、摘要交给便宜模型或本地工具，架构设计和跨文件修复再用强模型。
  • 为常见提示和上下文做模板化，减少每次把无关文件塞进上下文的 token 浪费。
  • 建立小型 eval：同一组真实任务在不同模型、不同 effort 下比较成功率、耗时和成本。
  • 现在可做一步：梳理最近 20 次 AI 编程任务，把“必须强模型”和“可降级任务”分开标记。

Say they(OpenAi Etc)don’t find a way to reduce the cost of running these LLMs. Will we shift towards slower/worse LLMs running locally? Or maybe enterprise ones only used by large corporations for specific tasks?

Will the era of using these to generate code end? Is the assume that the inference problem will be solved?
