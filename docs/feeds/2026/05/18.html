Ask HN 讨论 LLM 成本：把 AI 编程工作流设计成不怕模型涨价Ask HN on LLM Costs: Build AI Coding Workflows That Survive Model Price Changes

把 AI 编程任务分层路由，降低对低价强模型的依赖。

Route AI coding tasks by tier to reduce dependence on cheap frontier models.

这类关于“低价 LLM 是否只是短暂窗口”的讨论，核心提醒是：不要把 AI 编程工作流设计成只能依赖无限便宜的 frontier model；要把任务按价值、风险和上下文量分层路由。

  • 高价值任务用强模型：架构设计、复杂 bug、跨文件重构、安全审查、最终 code review。
  • 低价值或重复任务用便宜路径：格式化、简单测试修复、日志归纳、grep 后摘要、样板代码生成。
  • 把上下文当预算管理：先让工具收集最小证据，再让模型推理；避免把整个仓库无差别塞进 prompt。
  • 建立“模型降级也可用”的提示词：要求 agent 明确假设、列出需要读取的文件、先给计划再改代码。
  • 把昂贵推理放在流程关键点，而不是每一步都开最高档：例如“先 cheap scan，再 expensive review”。
  • 本地模型和小模型更适合做预处理：分类 issue、生成候选文件列表、压缩日志、整理测试失败；不要指望它们独立完成高风险跨模块改动。

可以直接套用一个路由规则：

Use the cheapest capable path first.
1. For search, file listing, formatting, and log summarization, use tools and small/cheap models.
2. Escalate to the strongest coding model only when the task needs cross-file reasoning, API design, security judgment, or final review.
3. Before escalating, produce a compact evidence packet: files read, failing commands, observed errors, and the exact decision needed.

Discussions about whether low-cost LLMs are temporary are a useful reminder: do not build AI coding workflows that only work when frontier models are infinitely cheap. Route tasks by value, risk, and context size.

  • Use strong models for high-value work: architecture, complex bugs, cross-file refactors, security review, and final code review.
  • Use cheaper paths for low-value or repetitive work: formatting, simple test fixes, log summarization, grep-based summaries, and boilerplate generation.
  • Treat context as a budget: gather the minimum evidence with tools first, then ask the model to reason; avoid dumping the whole repo into the prompt by default.
  • Make prompts resilient to model downgrades: require explicit assumptions, file lists, and a plan before edits.
  • Spend expensive reasoning at key checkpoints, not on every step: for example, cheap scan first, expensive review later.
  • Local and small models are better for preprocessing: issue classification, candidate file lists, log compression, and test-failure clustering; do not rely on them alone for high-risk cross-module changes.
