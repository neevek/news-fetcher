Claude Code v2.1.152：`/code-review --fix` 自动落地评审修改，新增 Skills 热重载与工具权限治理Claude Code v2.1.152: Auto-Apply Review Fixes, Hot-Reload Skills, and Stronger Tool Governance

评审可自动改代码，技能与治理能力同步增强。

Auto-applied review fixes plus stronger skills and governance controls.

这是 Claude Code v2.1.152 的一次高价值工作流更新，核心是把“评审建议→自动落地修改”打通，并增强了 Skills/Hook/插件治理能力。对日常使用 Claude Code 与 Codex 的工程师来说，重点是把它接入你的代码评审、技能热更新和团队安全边界。

  • 用 /code-review --fix 直接把评审建议落到工作区，减少“看完建议还要手动改”的往返。
    • 适合场景：重构、重复代码清理、可读性优化、局部性能改进。
    • 建议流程：先在功能分支提交基线，再执行自动修复并做 diff 审阅。
    • 可直接执行：
# 1) 先保存当前状态
git add -A && git commit -m "baseline before review-fix"

# 2) 运行评审并自动应用建议
/code-review --fix

# 3) 检查改动并回归测试
git diff
npm test

  • /simplify 现在会调用 /code-review --fix，可以把“简化代码”变成一个稳定入口。

    • 实操建议：把 /simplify 作为“提交前清理”固定步骤，用于降低复杂度和重复逻辑。

  • Skills 和 slash commands 支持在 frontmatter 里配置 disallowed-tools，可在技能激活时临时移除工具权限。

    • 这对团队很关键：可以给高风险任务（如生产环境操作、外网访问）做最小权限运行。
    • 示例（按你的技能文件格式调整）：
---
name: safe-refactor
disallowed-tools:
  - web.search_query
  - exec_command
---

  • 新增 /reload-skills，无需重启会话即可重扫技能目录。

    • 典型收益：你在迭代 Skill 提示词或工具编排时，调试周期明显缩短。
    • 推荐循环：改 SKILL.md → /reload-skills → 立即回归测试。

  • SessionStart hook 可返回 reloadSkills: true，让 hook 安装的技能在同一会话立即可用。

    • 这适合“按仓库自动装配技能”的团队工作流：新人进入项目即可拿到对应技能集。

  • SessionStart hook 现在可通过 hookSpecificOutput.sessionTitle 设置会话标题（启动与恢复时）。

    • 可用于自动命名规范：repo/branch/task，方便后续检索与审计。

  • 新增 MessageDisplay hook 事件，可在展示层转换或隐藏助手消息。

    • 可落地用途：
    • 自动脱敏（屏蔽 token、密钥模式串）
    • 统一输出格式（例如把建议自动转成团队模板）
    • 合规拦截（阻止特定内容显示）

  • 新增 pluginSuggestionMarketplaces 托管设置，管理员可 allowlist 可被上下文建议引用的组织 marketplace。

    • 这让“插件推荐”从开放式变成可治理模式，适合企业内控和供应链风险控制。

  • claude plugin marketplace remove 增加 --scope user|project|local，与 add/install/uninstall 语义对齐。

    • 直接用法：
claude plugin marketplace remove <marketplace-id> --scope project
  • 版本还提到会话内切换到 --fallback-model 的行为改进（用于剩余会话）。
    • 实操建议：为长会话明确配置 fallback，避免主模型不可用时中断流水线。
claude --model <primary-model> --fallback-model <backup-model>

现在就能做的一步：把 /code-review --fix 加进你的提交前流程（本地脚本或团队文档），并为高风险 Skill 增加 disallowed-tools 最小权限配置。

Claude Code v2.1.152 is a high-impact workflow release: it closes the loop from “review findings” to “applied code changes,” while upgrading Skills/Hook/plugin governance. For Claude Code and Codex users, the practical value is faster review cycles, hot-reloadable skills, and tighter team safety controls.

  • Use /code-review --fix to apply review findings directly to your working tree.
    • Best for refactors, deduplication, readability cleanups, and local efficiency improvements.
    • Recommended flow: commit a baseline, run auto-fix, then review diffs.
    • Run this now:
# 1) Save a baseline
git add -A && git commit -m "baseline before review-fix"

# 2) Run review + auto-apply findings
/code-review --fix

# 3) Inspect and validate
git diff
npm test

  • /simplify now invokes /code-review --fix, giving you one stable entry point for cleanup.

    • Practical habit: make /simplify a pre-commit cleanup step to reduce complexity and repeated logic.

  • Skills and slash commands can now set disallowed-tools in frontmatter to remove tools while a skill is active.

    • This is important for teams: apply least-privilege boundaries on risky tasks (prod ops, web access, etc.).
    • Example (adapt to your skill format):
---
name: safe-refactor
disallowed-tools:
  - web.search_query
  - exec_command
---

  • New /reload-skills rescans skill directories without restarting the session.

    • Immediate benefit: shorter debug loops while iterating skill prompts/tool orchestration.
    • Suggested loop: edit SKILL.md → /reload-skills → test immediately.

  • SessionStart hooks can now return reloadSkills: true, making hook-installed skills available in the same session.

    • Great for repo-aware onboarding: load project-specific skills automatically on session start.

  • SessionStart hooks can set a session title via hookSpecificOutput.sessionTitle on start/resume.

    • Use it for naming conventions like repo/branch/task for better search and auditability.

  • New MessageDisplay hook event can transform or hide assistant text at display time.

    • Direct uses:
    • auto-redaction (tokens/secrets patterns)
    • standardized output formatting
    • compliance filters before rendering

  • New managed setting pluginSuggestionMarketplaces lets admins allowlist org marketplaces for context-aware plugin suggestions.

    • This turns plugin suggestions into a governed channel, useful for enterprise controls and supply-chain risk reduction.

  • claude plugin marketplace remove now accepts --scope user|project|local, aligned with add/install/uninstall.

    • Example:
claude plugin marketplace remove <marketplace-id> --scope project
  • The release also notes improved session behavior around switching to --fallback-model for the remainder of a session.
    • Practical move: define a fallback model for long-running sessions to avoid pipeline interruption.
claude --model <primary-model> --fallback-model <backup-model>

One step to do now: add /code-review --fix to your pre-commit workflow and enforce least-privilege via disallowed-tools on high-risk skills.

Back to blog☀ LightMay 20, 2026The Correctness LayerAIData EngineeringAltimateWe started altimate-code with one goal: build a harness that data engineers could actually trust with their pipelines and platforms. The first thing that became clear is that LLMs are the wrong tool for half of that job. Two queries either are or are not semantically equivalent. A lineage edge either does or does not exist. A row-level diff is either correct or not. None of those are creative questions, and a probability distribution is the wrong answer to any of them. Over the past year, we have built a three-layer architecture that puts the LLM in charge of the work it is good at (strategy, intent-parsing, code generation) and a deterministic Rust and TypeScript stack underneath for everything that has to be reproducible. The split (detailed later) is what gets us to the top of the ADE and the DAB benchmarks, and more importantly, what makes the result repeatable on someone else's hardware. This post walks through how the split actually works: where the boundary sits, the three operations we moved out of the model entirely, and what thousands of structured benchmark traces taught us about where agents actually fail. LLMs are probability distributions Run the same prompt through the same model twice, and you'll get two different answers. That's not a bug. It's the substrate. Language models sample from a probability distribution every time they generate a token, and the distribution is shaped — sometimes broadly — by sampling temperature[1], prompt-cache state, and the internal nondeterminism of large batched inference on shared hardware[2]. For creative work, this is exactly what you want. Drafting a SQL query, summarizing a result, recovering from an unexpected error — these are open-ended tasks with many acceptable answers, and probabilistic output is what makes the model useful. For correctness, it's a liability. "Are these two queries semantically equivalent?" has one right answer. "What columns does this output project from this source?" has one right answer. "Are these two tables byte-equal?" has one right answer. When a probabilistic system answers a deterministic question, you can't cache the answer (it might be different next time), you can't debug it (a single sample tells you nothing about the distribution), and you can't benchmark it (a single pass-rate is a coin flip away from a different number). What this looks like on a real benchmark We took one task off the ADE benchmark, a dbt refactor called asana004, and ran it through altimate-code three times with the same model, the same prompt, and the same starting state. The agent passed it once and failed it twice. Across the three runs, dbt's test counts came back as 6/6, then 4/6, then 5/6. If you are building a data-engineering agent, that is the problem you have to design around. You cannot trust a system whose output is intrinsically random. You cannot cache it.
