Codex 如何在 Windows 上做 sandbox：专用用户、restricted token 与 firewall 网络控制How Codex Sandboxes Windows: Dedicated Users, Restricted Tokens, and Firewall-Based Network Control

理解 Codex Windows sandbox，能更安全地配置联网与写权限。

Understanding the Codex Windows sandbox helps configure network and write permissions safely.

OpenAI 这篇 Windows sandbox 工程文章解释了 Codex 如何在 Windows 上同时做到“能干活”和“受限制”：不是简单依赖 Windows Sandbox，而是组合专用用户、restricted token、ACL 和 firewall rules 来约束 agent 执行的命令。

  • Codex 默认需要限制两件事：只能在允许位置写文件，且网络访问需要受控；否则 coding agent 在本机执行命令的风险会过高。
  • AppContainer、Windows Sandbox、Mandatory Integrity Control 都有形状不匹配的问题：要么太窄，要么脱离真实 checkout，要么会改变宿主文件系统信任语义。
  • 早期 unelevated 方案用 synthetic SID 和 write-restricted token 控制写入，并用 proxy 环境变量阻断常见网络工具，例如 HTTPS_PROXY=http://127.0.0.1:9、GIT_SSH_COMMAND=cmd /c exit 1；这对普通工具有效，但无法阻止直接 socket 访问。
  • 当前 elevated sandbox 会创建专用本地用户：CodexSandboxOffline 用 firewall rules 阻断出站网络，CodexSandboxOnline 用于允许联网场景；这比只改环境变量更接近真实安全边界。
  • 架构拆成 codex.exe、codex-windows-sandbox-setup.exe、codex-command-runner.exe 和 child process；setup 需要提升权限，但日常 Codex harness 可以保持非提升运行。
  • 对工程师的实际含义：在 Windows 上用 Codex 时，优先使用默认 sandbox/approval 工作流，不要为了省确认长期打开 Full Access。
  • 如果任务需要安装依赖、访问 registry 或调用外网 API，让 Codex明确请求联网步骤；平时让测试、搜索、编辑在 offline sandbox 中完成。
  • 对内部 agent 工具开发也有启发：安全边界应由 OS enforcement 实现，环境变量级别的“禁止联网”只能作为辅助手段。

OpenAI’s Windows sandbox engineering write-up explains how Codex stays both useful and constrained on Windows: it does not simply rely on Windows Sandbox, but composes dedicated users, restricted tokens, ACLs, and firewall rules to constrain commands launched by the agent.

  • Codex needs to constrain two things by default: file writes should only happen in allowed locations, and network access should be controlled; otherwise local command execution by a coding agent is too risky.
  • AppContainer, Windows Sandbox, and Mandatory Integrity Control each failed to fit the workload: too narrow, detached from the real checkout, or changing trust semantics on the host filesystem.
  • The early unelevated design used a synthetic SID and write-restricted token for file writes, plus proxy environment variables to block common network tools, such as HTTPS_PROXY=http://127.0.0.1:9 and GIT_SSH_COMMAND=cmd /c exit 1; this works for normal tools but cannot stop direct socket access.
  • The current elevated sandbox creates dedicated local users: CodexSandboxOffline is targeted by firewall rules that block outbound network access, while CodexSandboxOnline is used for allowed-network scenarios; this is a stronger boundary than environment variables alone.
  • The architecture is split into codex.exe, codex-windows-sandbox-setup.exe, codex-command-runner.exe, and the child process; setup needs elevation, but the day-to-day Codex harness can remain unelevated.
  • Practical takeaway: on Windows, prefer the default sandbox/approval workflow and avoid leaving Full Access enabled just to reduce prompts.
  • If a task needs dependency installation, registry access, or an external API, make Codex request that networked step explicitly; keep tests, search, and edits in the offline sandbox by default.
  • For internal agent-tool builders, the broader lesson is that security boundaries should be OS-enforced; environment-variable network blocking is only a helper, not a robust control.

用 Codex 搭建“评审-修复-验证”的自动修复循环Build a Review-Repair-Validate Loop with Codex

把测试反馈变成下一轮 Codex 修复输入。

Turn test feedback into the next Codex repair input.

OpenAI Cookbook 这篇示例把 Codex 当成可脚本化的修复执行器，用 Review -> Repair -> Validate 的闭环来更新过期 API / SDK 示例。对 Codex 用户最有价值的点是：不要只让 AI “改一下”，而是把检查结果、测试失败和剩余问题结构化地喂回下一轮。

  • 核心模式可以直接搬到代码维护、文档更新、迁移脚本和测试修复：
    • Review：只读检查，输出结构化 findings，不改文件。
    • Repair：基于 findings 和上一轮 validation delta 做最小改动。
    • Validate：跑测试、lint、schema 校验、notebook 执行或人工规则评分，把失败转成下一轮输入。
  • 用 codex exec 的关键是强制结构化输出，避免后续脚本解析自然语言：使用 --output-schema 和 --output-last-message 让每轮都写出可审计的 JSON。
  • 给 Codex 一个明确的“业务规则合同”，比让它自由推断更稳。例如文档迁移可以显式写入当前 API 形态、允许改动范围、必须保留的教学目标、运行环境要求。
  • 验证必须跑在修复之后，而不是让模型自称已经修好。Notebook 示例用 jupyter nbconvert --execute；代码库里可以替换成 pytest、npm test、cargo test、ruff check、tsc --noEmit 或自定义 eval。
  • 每轮保留 record.json：包含 review findings、Codex 改动摘要、验证结果、remaining_delta。这会把 AI 修复从“看起来改了”变成可复盘的维护流程。
  • 生产循环要设置停止条件：验证通过、达到最大轮数、remaining_delta 不再变化、需要人类判断。没有停止条件的 agent loop 很容易浪费 tokens 或反复改同一处。
  • 多个互不相关的 artifact 可以并发跑：文档、notebook、测试 fixture、迁移案例都可以拆成独立 case，用线程池或任务队列并行执行。

可以先用下面这个最小骨架改造你自己的仓库，把 validate() 换成项目真实检查命令：

import json
import os
import shlex
import subprocess
from pathlib import Path

CODEX = 'codex'
MODEL = os.getenv('REPAIR_MODEL', 'gpt-5.4-mini')
EFFORT = os.getenv('REPAIR_REASONING_EFFORT', 'low')

review_schema = {
    'type': 'object',
    'properties': {
        'findings': {
            'type': 'array',
            'items': {
                'type': 'object',
                'properties': {
                    'file': {'type': 'string'},
                    'issue_type': {'type': 'string'},
                    'severity': {'type': 'string'},
                    'description': {'type': 'string'},
                    'suggested_fix': {'type': 'string'}
                },
                'required': ['file', 'issue_type', 'severity', 'description', 'suggested_fix'],
                'additionalProperties': False
            }
        }
    },
    'required': ['findings'],
    'additionalProperties': False
}

fix_schema = {
    'type': 'object',
    'properties': {
        'changes_made': {'type': 'array', 'items': {'type': 'string'}},
        'unresolved_items': {'type': 'array', 'items': {'type': 'string'}}
    },
    'required': ['changes_made', 'unresolved_items'],
    'additionalProperties': False
}

def run_codex_json(prompt: str, schema: dict, run_dir: Path) -> dict:
    run_dir.mkdir(parents=True, exist_ok=True)
    schema_file = run_dir / 'schema.json'
    answer_file = run_dir / 'answer.json'
    schema_file.write_text(json.dumps(schema, indent=2), encoding='utf-8')

    cmd = f'''{CODEX} exec
      --model {MODEL}
      --sandbox workspace-write
      --ask-for-approval never
      --config model_reasoning_effort={EFFORT}
      --output-schema {schema_file}
      --output-last-message {answer_file}
      -'''
    result = subprocess.run(shlex.split(cmd), input=prompt, text=True, capture_output=True)
    if result.returncode != 0:
        raise RuntimeError(result.stderr)
    return json.loads(answer_file.read_text(encoding='utf-8'))

def validate() -> dict:
    result = subprocess.run(['pytest', '-q'], text=True, capture_output=True)
    return {
        'passed': result.returncode == 0,
        'feedback': (result.stdout + result.stderr)[-4000:]
    }

def repair_until_done(target_files: list[str], max_iterations: int = 3):
    remaining_delta = []
    for i in range(1, max_iterations + 1):
        review_prompt = f'Review these files without editing them: {target_files}. Return focused findings.'
        review = run_codex_json(review_prompt, review_schema, Path(f'.codex-loop/{i}/review'))

        repair_prompt = f'''Repair the smallest useful set of issues.
Files: {target_files}
Review findings: {json.dumps(review, indent=2)}
Validation feedback from previous pass: {json.dumps(remaining_delta, indent=2)}
Do not claim success; validation will decide.'''
        repair = run_codex_json(repair_prompt, fix_schema, Path(f'.codex-loop/{i}/repair'))
        validation = validate()

        record = {'review': review, 'repair': repair, 'validation': validation}
        Path(f'.codex-loop/{i}/record.json').write_text(json.dumps(record, indent=2), encoding='utf-8')

        if validation['passed']:
            break
        remaining_delta = [validation['feedback']]

现在能做的一步：选一个经常需要 AI 反复修的任务，比如“修到 pytest -q 通过”或“把旧 SDK 示例迁到新 API”，先只接入一个验证命令，再让 Codex 每轮读取上一轮 record.json 的 remaining_delta。

This OpenAI Cookbook example treats Codex as a scriptable repair executor and uses a Review -> Repair -> Validate loop to refresh stale API / SDK examples. The useful lesson for Codex users is: do not just ask AI to “fix it”; feed structured findings, test failures, and remaining deltas into the next pass.

  • The core pattern applies directly to code maintenance, docs refreshes, migration scripts, and test repair:
    • Review: inspect only, return structured findings, do not edit files.
    • Repair: make the smallest useful change using findings plus the previous validation delta.
    • Validate: run tests, lint, schema checks, notebook execution, or rule-based judging, then turn failures into next-pass input.
  • The key codex exec move is structured output: use --output-schema and --output-last-message so every pass emits auditable JSON instead of prose that later scripts must scrape.
  • Give Codex a clear business-rule contract instead of asking it to infer everything. For a docs migration, spell out the current API shape, allowed edit scope, teaching goals to preserve, and runtime requirements.
  • Validation must run after repair; do not let the model declare success. The notebook example uses jupyter nbconvert --execute; in a codebase, swap in pytest, npm test, cargo test, ruff check, tsc --noEmit, or a custom eval.
  • Keep a record.json for each pass: review findings, Codex change summary, validation result, and remaining_delta. That turns AI repair from “it seems changed” into a reviewable maintenance workflow.
  • Production loops need stop conditions: validation passes, max attempts reached, remaining_delta stops changing, or human judgment is required. Without a stop condition, an agent loop can burn tokens or churn on the same issue.
  • Independent artifacts can run concurrently: docs, notebooks, test fixtures, and migration cases can each be separate cases processed with a thread pool or job queue.
