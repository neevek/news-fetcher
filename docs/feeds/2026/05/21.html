  • 关键能力：通过本地 MCP 通道向代理暴露“可执行操作”，而不是暴露凭证明文。
  • 工程收益：减少 .env 散落、提交泄漏、日志外溢，同时保留自动配置环境的速度。
  • 操作模式：每次敏感访问都要求用户批准，适合把人类审批嵌入代理执行链路。
  • 推荐实践：把“创建环境、列出变量名、运行依赖密钥的任务”交给代理；把“查看/导出密钥值”留在密管侧。
  • 与 Claude Code 的通用启发：无论供应商，优先采用“工具拿权限、模型不见密钥”的体系。

可直接执行的迁移步骤：

1) 清点仓库与本地 .env 中的长期密钥
2) 迁移到 1Password（或同类 secret manager）
3) 在 AI 编码流程中仅保留变量引用，不保留真实值
4) 为高风险操作加“必须审批”策略
5) 在 CI 增加 secret 扫描，防回流到代码库

参考：https://1password.com/blog/1password-trusted-access-layer-for-openai-codex

The core of this release is integrating 1Password MCP Server with Codex to enforce an access model where the agent can act without owning raw secrets. For Codex / Claude Code users, this is a practical security workflow template you can adopt now.

  • Key capability: expose executable actions over local MCP, not credential plaintext.
  • Engineering benefit: less .env sprawl, fewer commit leaks, less log exposure, while keeping environment setup fast.
  • Operating mode: sensitive access requires explicit user approval, embedding human gates into agent execution.
  • Recommended split: let agents create environments, list variable names, and run secret-dependent tasks; keep secret value viewing/export strictly in the vault layer.
  • Cross-tool lesson for Claude Code too: prefer architectures where tools get permissions and models never see raw secrets.

Direct migration steps:

1) Inventory long-lived secrets in repos and local .env files

3) Keep only variable references in AI coding workflows
4) Add mandatory approval for high-risk operations

建议立刻调整团队用法：

旧流程：/simplify -> 直接改代码
新流程：/code-review high -> 人工确认 -> 再执行修复动作

PR 审查接入示例：

  • Pinned background sessions (Ctrl+T) now stay alive when idle, restart in place after updates, and are de-prioritized for shedding under memory pressure.
  • /simplify is renamed to /code-review with effort levels (for example /code-review high) focused on correctness bugs.

这是典型的可用性抱怨帖，但对高频用户很有现实意义：连接稳定性直接决定 AI 编程的实际产能。对 Claude Code / Codex 用户来说，最该做的是把“断连体验”工程化记录并绕开高损耗路径。

  • 建立断连日志：记录时间、任务类型、仓库体量、是否长输出、是否并发操作，便于识别触发模式。
  • 减少长会话脆弱性：把大任务拆成可提交的小步骤，每步都落盘到 PLAN.md / TODO.md。
  • 降低上下文重建成本：关键约束写进固定提示模板，断线后可快速恢复。
  • 结果优先策略：先让代理生成 patch/PR，再做解释；避免把时间耗在超长对话上下文。
  • 多工具兜底：主工具不稳时，切到备用链路继续推进，避免整段开发阻塞。

可直接套用的恢复提示词：

你刚刚断线重连。请先读取 `PLAN.md` 和最近一次 git diff，
1) 用 5 行总结当前状态；
2) 只继续未完成步骤；
3) 每完成一步就提交最小可审查改动。

参考：https://news.ycombinator.com/item?id=48217688

This is a typical reliability complaint thread, but it matters operationally: session stability directly impacts real AI coding throughput. For Claude Code / Codex users, the practical move is to instrument reconnect pain and route around high-friction patterns.

  • Keep a reconnect log: timestamp, task type, repo size, long-output usage, and concurrency to identify trigger patterns.
  • Reduce long-session fragility: split work into small, commit-ready steps and persist state in PLAN.md / TODO.md.
  • Lower context rebuild cost: keep hard constraints in a fixed prompt template for fast recovery.
  • Optimize for artifacts first: have the agent produce patches/PRs early, then ask for explanations.
  • Maintain a fallback tool path: when one tool is unstable, switch execution path to avoid blocking delivery.

Recovery prompt you can reuse:
