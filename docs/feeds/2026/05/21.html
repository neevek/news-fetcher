1Password Environments MCP Server 让 Codex 用密钥但看不到密钥1Password Environments MCP Server Lets Codex Use Secrets Without Seeing Them

Agentic 编程的密钥治理开始进入工作流。

Secrets governance is moving directly into agentic coding workflows.

OpenAI 与 1Password 的 Codex 集成 把密钥管理放进 agentic coding workflow：Codex 可以通过 1Password Environments MCP Server 使用凭据，但密钥值不进入 prompt、代码、终端输出或模型上下文。对工程团队来说，这是让 AI agent 参与真实数据库、API、部署流程时更可审计的做法。

  • 核心架构是本地 MCP server 连接 Codex 与 1Password Environments；1Password desktop app 负责身份、授权和访问批准。
  • Codex 可以创建环境、管理变量名、准备配置、运行需要密钥的应用；密钥值由 1Password 在运行时注入到授权进程，Codex 不读取原始值。
  • 每次访问都需要用户显式批准，团队可用 1Password 的权限和策略控制哪些人、哪些 agent、哪些环境可用。
  • 适合优先落地的场景：新项目 bootstrap 环境变量、把现有 .env/脚本/硬编码密钥迁进 1Password、让 Codex 扫描 repo 中的明文 secret 并替换为受管引用、从 local 扩展到 staging/production 环境模板。
  • 使用 1Password Enterprise Password Manager 的组织需要启用 1Password Environments beta policy；个人用户需要在 1Password Labs 中 opt in，并通过 1Password Marketplace 文档连接 Codex。
  • 这类集成的趋势很明确：agent 需要访问真实系统，但不应拥有密钥本身。以后评估 AI 编程工具时，要把“secret 是否进入模型上下文”列为安全红线。

可直接给 Codex 的迁移提示词：

Use the 1Password Environments MCP server for this repo.

Goal:
- Find plaintext secrets or env values in `.env*`, scripts, CI examples, README snippets, and source files.
- Propose a migration plan before editing.
- Move secrets into 1Password-managed environments when I approve.
- Replace code/config usage with environment variable references, not raw values.

Constraints:
- Never print secret values.
- Never write secret values to disk.
- Ask for approval before granting access to any 1Password environment.
- Keep local, staging, and production variables separate.

Return:
- Files inspected
- Secrets or risky patterns found, with values redacted
- Proposed variable names
- Files to change
- Test command to verify the app still starts

Codex 0.133.0：Goals、权限配置与插件发现更适合团队工作流Codex 0.133.0 Turns Goals, Permissions, and Plugins Into Team-Ready Workflow Primitives

Goals 和权限配置让 Codex 更适合团队化长任务。

Goals and permission profiles make Codex stronger for team-scale long-running work.

Codex 0.133.0 是一次偏工作流和平台能力的版本更新：Goals、remote-control、permission profiles、plugin discovery、extension lifecycle 都更适合长期任务、团队化配置和自动化集成。对 Codex 用户来说，重点不是“多了几个功能”，而是可以把以前靠口头约定和临时脚本维持的 agent 工作流，收进可观察、可配置、可复用的工具链。

  • Goals 默认启用，并有专用存储，可跨 active turns 追踪进度：更适合把一个大任务拆成持续目标，而不是每轮都重新提醒 agent。
  • codex remote-control 现在更像前台命令：会等待 readiness、报告 machine status，同时保留显式 daemon 风格的 start / stop。这对远程工作区、CI 预热、开发机常驻控制都更稳。
  • permission profiles 增强了 list API、继承、托管 requirements.toml、运行时刷新，以及更强的 Windows sandbox 集成：团队可以把“哪些命令可运行、哪些资源可访问”做成可审计配置。
  • plugin discovery 输出更可检查：能看到 marketplace-aware list、installed versions、marketplace roots、remote collection，有助于排查“为什么这个插件没被发现/版本不对”。
  • extensions 可观察更多生命周期事件，包括 subagent start/stop、tool execution、turn metadata、async approval/turn processing：可以开始做更细的审计、指标、成本统计和内部治理。
  • 修复点里值得关注：TUI 复用 local app-server socket 时选错工作目录、Plan mode 中 Shift+Enter 意外提交、Windows PowerShell / sandbox 相关问题。这些都直接影响日常交互稳定性。

现在可以做的一步：升级后先把本机能力盘点出来，建立一份团队可复现的 Codex 环境快照。

# 检查权限 profile，可用子命令以本机 help 为准
codex permissions --help

实践建议：

  • 对长期任务，让 Codex 使用 Goals 记录阶段目标，例如“先复现、再定位、再最小修复、最后验证”。
  • 对团队环境，把 requirements.toml 和 permission profile 一起评审，避免每个人本地 agent 权限不同导致结果不可复现。
  • 对插件问题，先看 codex plugin list 的 marketplace roots 和 installed versions，再排查路径或版本。
  • 对 Windows 用户，优先验证 PowerShell、sandbox、worktree、background session 这些过去容易出问题的路径。

Codex 0.133.0 is a workflow and platform-focused release: Goals, remote control, permission profiles, plugin discovery, and extension lifecycle hooks all become more useful for long-running tasks, team configuration, and automation. For Codex users, the real value is moving fragile conventions and one-off scripts into observable, configurable, reusable agent workflows.

  • Goals are now enabled by default, backed by dedicated storage, and can track progress across active turns, which is useful when a large task needs durable structure instead of repeated reminders.
  • codex remote-control now behaves more like a foreground command: it waits for readiness, reports machine status, and still keeps explicit daemon-style start / stop commands. That helps with remote workspaces, CI warmup, and persistent developer-machine control.
  • Permission profiles gained list APIs, inheritance, managed requirements.toml, runtime refresh, and stronger Windows sandbox integration. Teams can now turn “which commands and resources may the agent use?” into auditable configuration.
  • Plugin discovery is easier to inspect through marketplace-aware list output, installed versions, visible marketplace roots, and remote collection support. This helps debug “why was this plugin not discovered?” or “why is the wrong version active?”
  • Extensions can observe more lifecycle events, including subagent start/stop, tool execution, turn metadata, and async approval/turn processing. That opens the door to better auditing, metrics, cost tracking, and internal governance.
  • The fixes worth noting include incorrect TUI working directory selection when reusing a local app-server socket, Shift+Enter unexpectedly submitting in Plan mode, and Windows PowerShell / sandbox issues. These directly affect day-to-day interaction stability.

  • For plugin problems, first inspect marketplace roots and installed versions through codex plugin list, then debug path or version issues.

Claude Code v2.1.147 强化后台 agent、`/code-review` 与企业策略Claude Code v2.1.147 Strengthens Background Agents and Reframes `/code-review`

后台任务更稳，代码审查命令语义变了。

Background work is sturdier, and the review command changed meaning.

Claude Code v2.1.147 是一次工作流型更新：后台 agent 更稳定，/simplify 被重新定义为 /code-review，企业登录策略和多处终端/Windows/MCP 边界问题得到修复。对日常用 Claude Code 跑长任务和做 PR review 的团队，影响很直接。

  • Pinned background sessions 更适合长任务：在 claude agents 里用 Ctrl+T pin 的后台会话，空闲时会保持存活；Claude Code 更新时会原地重启以应用更新；内存压力下也会优先保留 pinned sessions。
  • /simplify 已改为 /code-review，语义从“清理并修复”变成“按 effort 查 correctness bugs”。如果你原来把 /simplify 当自动重构命令，需要更新个人习惯和团队文档。
  • /code-review high 适合在 PR 前跑一次高强度 review；/code-review high --comment 可把 findings 作为 GitHub PR inline comments 发布，适合 reviewer bot 或 release 前检查。
  • Auto-updater 现在会重试临时网络错误，并报告具体错误类别、OS error code 和当前版本；这能减少“更新失败但不知道为什么”的排障时间。
  • 大文件编辑的 diff rendering 更快，长 diff review 和 agent 批量修改时体感会更好。
  • Prompt history 不再连续记录重复项，反复 arrow-up 提交同一 prompt 时历史更干净。
  • 企业安全修复值得尽快升级：forceLoginOrgUUID 和 forceLoginMethod managed settings 现在会约束 third-party-provider 与 API-key sessions。
  • Headless/SDK 模式下未知 slash command 不再静默失败，而是显示错误；这能让自动化脚本更早暴露拼写或版本兼容问题。
  • 其他高频修复包括：! command 输出中的 & 不再变成 &amp;、MCP server 分页不再丢 page 1 之后的 resources/templates/prompts、pasted text 不再变成不可读占位符、plugin agents 多个 Agent(...) tools 不再只保留最后一个。
  • Windows/PowerShell 用户也应升级：修复了 PowerShell 输出丢失、winget/Microsoft Store 安装 pwsh 时工具失败、权限“don't ask again”规则不匹配、Windows Terminal attached background session strobing、NTFS junction 删除风险等问题。

需要马上改的用法：

# Before
/simplify

# After: correctness review
/code-review medium
/code-review high

# Post findings as GitHub PR inline comments
/code-review high --comment

建议把 PR 前检查流程改成：

Run /code-review high on this branch. Focus on correctness bugs, missed edge cases, security-sensitive behavior, and tests that should have been updated. Do not make code changes; report findings first with file and line references.

如果你依赖后台长任务，可以建立固定操作：先启动 agent team，再对关键 session 使用 Ctrl+T pin，最后要求 lead agent 持续写进度日志，避免更新或空闲回收打断任务。

Claude Code v2.1.147 is a workflow-heavy update: background agents are more durable, /simplify has been redefined as /code-review, and several enterprise login, terminal, Windows, and MCP edge cases are fixed. For teams using Claude Code for long-running tasks and PR review, the impact is immediate.

  • Pinned background sessions are better for long tasks: sessions pinned with Ctrl+T in claude agents stay alive while idle, restart in place to apply Claude Code updates, and are shed under memory pressure only after non-pinned sessions.
  • /simplify is now /code-review, and its meaning changed from “clean up and fix” to “report correctness bugs at an effort level.” If you used /simplify as an auto-refactor command, update personal habits and team docs.
  • /code-review high is a good pre-PR deep review; /code-review high --comment can post findings as GitHub PR inline comments, which fits reviewer bots or pre-release checks.
  • The auto-updater now retries transient network failures and reports specific error categories, OS error codes, and the current version. That reduces time spent debugging vague update failures.
  • Diff rendering is faster for large file edits, which helps when reviewing long diffs or agent-generated bulk changes.
  • Prompt history no longer stores consecutive duplicates, so repeated arrow-up submissions keep history cleaner.
  • The enterprise security fix is worth upgrading for: forceLoginOrgUUID and forceLoginMethod managed settings now apply to third-party-provider and API-key sessions.
  • Unknown slash commands in headless/SDK mode now show an error instead of failing silently, so automation catches spelling or version compatibility problems earlier.
  • Other common fixes include: & in ! command output no longer becomes &amp;, MCP server pagination no longer drops resources/templates/prompts after page 1, pasted text no longer turns into unreadable placeholders, and plugin agents declaring multiple Agent(...) tools no longer keep only the last one.
  • Windows/PowerShell users should upgrade too: fixes cover dropped PowerShell output, failures when pwsh is installed through winget or the Microsoft Store, broken “don't ask again” permission rules, Windows Terminal strobing in attached background sessions, and NTFS junction risk when removing background-job worktrees.

Update these habits immediately:

# After: correctness review
/code-review medium
/code-review high

# Post findings as GitHub PR inline comments
/code-review high --comment

A better pre-PR check prompt:
