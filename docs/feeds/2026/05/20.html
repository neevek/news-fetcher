Stainless 之后怎么办：为 OpenAPI→SDK 生成链建立可替代方案After Stainless: How to De-Risk Your OpenAPI-to-SDK Pipeline

Stainless 变局提醒你：SDK 生成链必须去单点、可替代。

The Stainless shift is a reminder to de-risk SDK generation with portable alternatives.

这条讨论背后的关键变化是：Stainless 被 Anthropic 收购后，原本跨厂商共享的 SDK 自动生成基础设施出现收缩。对 Claude Code / Codex 用户，这意味着你需要把 SDK 供应链“内建化”，避免依赖单点厂商。

  • 先做资产保全：立刻归档当前已生成 SDK 源码、生成配置、OpenAPI 快照、发布流水线脚本。
  • 建立“可替代生成链”：以 OpenAPI Generator 或 Speakeasy 等方案做并行 PoC，验证主要语言产物质量。
  • 把 SDK 当产品维护：补齐版本策略、兼容性测试、变更日志与弃用策略，不再把它当一次性产物。
  • 为 AI 代理准备稳定接口：无论你用 Codex 还是 Claude Code，稳定、可预测的 SDK 比临时 HTTP 调用更利于自动化。
  • 组织层面启发：AI 时代的生产力瓶颈正从“模型能力”转向“工具链控制权与可迁移性”。

现在就能做的一步（最小迁移 PoC）：

# 以 OpenAPI Generator 为例（Java 版本）
docker run --rm -v ${PWD}:/local openapitools/openapi-generator-cli generate \
  -i /local/openapi.yaml \
  -g python \
  -o /local/sdk-python

# 生成后立刻做最小可用验证
cd sdk-python
pip install -e .
python -c "import <your_sdk_package>; print('ok')"

The key shift here is infrastructure concentration: after Anthropic acquired Stainless, a shared cross-vendor SDK generation layer appears to be shrinking. For Claude Code / Codex users, the practical implication is to internalize your SDK supply chain and remove single-vendor dependency.

  • Preserve assets now: archive generated SDK source, generation configs, OpenAPI snapshots, and release pipeline scripts.
  • Build an alternative generation path: run parallel PoCs with options like OpenAPI Generator or Speakeasy, and validate output quality in your target languages.
  • Treat SDKs as products: define versioning policy, compatibility tests, changelogs, and deprecation rules.
  • Give AI agents stable interfaces: whether using Codex or Claude Code, predictable SDKs beat ad-hoc HTTP calls for automation reliability.
  • Org-level lesson: in the AI era, productivity bottlenecks are shifting from model quality to toolchain control and portability.
