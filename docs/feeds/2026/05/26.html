为何 `/goal` 在复杂流程中会“失忆”：可落地的抗上下文腐化实践Why `/goal` Drifts on Complex Tasks and How to Stabilize It

长任务要外置状态与分阶段验证，防止上下文腐化。

For long tasks, externalize state and verify per phase to prevent context rot.

这类讨论指向一个现实问题：长流程任务里，/goal 可能因上下文压缩与记忆漂移导致计划与执行逐步失真。对 Claude Code/Codex 用户，重点不是停用功能，而是把任务拆解和状态外置。

  • 把“长期记忆”从模型上下文迁移到文件：PLAN.md（目标/里程碑）、STATE.md（当前决策/约束）、TODO.md（下一步可执行项）。
  • 每完成一个子任务就落盘，不依赖会话记忆连续性；下次调用先让模型读这 3 个文件再继续。
  • 避免一次性塞多目标：把复杂需求改成串行 checkpoint（每步都可验证、可回滚）。
  • 对每个阶段增加“重建上下文”提示词，强制模型先复述事实再动手，降低漂移。
  • 把“正确性检查”前移：每步都跑最小测试/静态检查，避免错误被带入后续压缩上下文。

可直接复用的提示词骨架：

你现在只执行一个阶段：{阶段名}。
先读取 PLAN.md、STATE.md、TODO.md，输出：
1) 你理解的当前目标（3 条）
2) 本阶段不应改变的约束（3 条）
3) 仅本阶段的执行清单
执行后：更新 STATE.md 和 TODO.md，并说明验证结果。

最小工作流：

# 每阶段循环
# 1) 让模型先“重建上下文”
# 2) 实施单一改动
# 3) 跑验证
# 4) 回写 STATE/TODO

This discussion highlights a practical risk: in long workflows, /goal can drift as context is compacted and memory quality decays. For Claude Code/Codex users, the key is not to abandon it, but to externalize state and decompose aggressively.

  • Move long-lived memory out of chat context into files: PLAN.md (milestones), STATE.md (current decisions/constraints), TODO.md (next executable steps).
  • Persist after every subtask; do not rely on session continuity. Start each run by reloading those three files.
  • Avoid multi-goal batches; run serial checkpoints where each step is verifiable and reversible.
  • Add a “context reconstruction” prompt at each phase so the model restates facts before editing, reducing drift.
  • Shift correctness checks left: run minimal tests/lint each step so errors do not propagate into later compressed context.

Reusable prompt skeleton:

Execute only one phase: {phase_name}.
First read PLAN.md, STATE.md, TODO.md and output:
1) your understanding of current goals (3 bullets)
2) constraints that must not change in this phase (3 bullets)
3) phase-only execution checklist
After execution: update STATE.md and TODO.md, then report verification results.

Minimal workflow loop:

# Per phase
# 1) Reconstruct context
# 2) Apply one focused change
# 3) Run verification
# 4) Write back STATE/TODO
